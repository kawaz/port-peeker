# DR-0002: 外部コマンドを使わず /proc を直接読む

- ステータス: Accepted
- 日付: 2026-05-06

## Context

設計書たたき台では「Phase 1 では `ss -tlnp` の出力をパース、Phase 2 で `/proc` ベースに置き換える」という段階計画になっていた。port LISTEN 判定とプロセス名解決の両方を、外部コマンド経由 (`ss`) で行うか、`/proc` を直接読むかを最初から決める必要があった。

## Decision

**最初から `/proc` 直読み一本で実装する。** `ss` などの外部コマンドには一切依存しない。

具体的な参照先:

- `/proc/net/tcp` および `/proc/net/tcp6` (LISTEN 行 = `st=0A` の inode を取得)
- `/proc/<pid>/fd/*` の readlink (`socket:[INODE]` 形式から pid を逆引き)
- `/proc/<pid>/comm` (プロセス名取得)

## Rationale

- **fork+exec のオーバーヘッドを排除**: LB ヘルスチェックは秒数十回来ることがある。リクエストごとに `ss` を起動すると累積コストが効く。`/proc` 直読みなら起動オーバーヘッドゼロ。
- **runtime 依存の削減**: `ss` は `iproute2` パッケージに含まれる。最小構成のコンテナ等では入っていない場合もある。`/proc` は Linux カーネルが提供するため必ず存在する。
- **権限要件は変わらない**: `/proc/net/tcp{,6}` は誰でも読める。`/proc/<pid>/fd` の readlink 制限 (他ユーザのプロセスは読めない) は `ss -tlnp` 経由でも同じ。`ss` を使っても回避できないので、外部コマンドを噛ませる利点がない。
- **実装量は同程度**: `/proc/net/tcp` は固定列のテキストフォーマットで、列を `strings.Fields` で分割するだけでパースできる。`ss` の出力パースと比較してコード量に差はない。むしろ `ss` のフォーマットは `iproute2` のバージョンによって微妙に変わる可能性があり、版差吸収の方が手間。
- **キャッシュとの相性**: TTL キャッシュで同一クエリは数秒に 1 回しか実 `/proc` 読み込みが走らない。fork+exec 削減と相まって低レイテンシを保てる。

## Alternatives Considered

- **`ss -tlnp` の出力パース**: exec オーバーヘッド、出力フォーマットの版依存、`iproute2` パッケージへの runtime 依存 (最小構成コンテナで欠落しがち)、いずれも本ツールの低レイテンシ・配布性ポリシーと噛み合わない。
- **netlink (`AF_NETLINK + sock_diag`)**: ss が内部で使っている API。最も低レベルかつ高速だが、メッセージ構築・解釈の実装コストが大きい。1 LB ヘルスチェック程度の頻度では `/proc` 直読みで十分速く、複雑性に見合う利点がない。
- **`gopsutil` などの Go ライブラリ**: `process.Processes()` 等で同等のことができるが、本ツールの用途 (port が LISTEN しているか + そのプロセス名) には機能過多。依存を増やすほどの価値はない。
