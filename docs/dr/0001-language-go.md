# DR-0001: 実装言語に Go を採用

- ステータス: Accepted
- 日付: 2026-05-06

## Context

kawaz の他リポ群 (`stable-which` / `cache-warden` / `authsock-warden` 等) は Rust 中心で実装している。一方、設計書たたき台 (`design.md`) では Go を提案している。port-peeker の責務は「LB のヘルスチェック HTTP リクエストを受け、`/proc` を読んで結果を返す」だけのワンバイナリツールである。実装言語を確定させる必要があった。

## Decision

**Go** を採用する。

## Rationale

- **静的リンク + クロスコンパイルが容易**: `CGO_ENABLED=0 GOOS=linux GOARCH=amd64 go build` で macOS から Linux バイナリが作れる。配布バイナリが 1 ファイルで完結する。
- **標準ライブラリで HTTP / TCP / TLS が完結**: `net/http`、`crypto/tls` が成熟しており、追加依存を持たずにヘルスチェック HTTP サーバを書ける。プロトコルチェック (将来) 実装時も標準ライブラリで賄える見込み。
- **メモリフットプリントが小さい**: 10MB 未満で常駐可能。LB ヘルスチェック専用エージェントとして許容範囲。
- **本ツールの責務は I/O 中心**: `/proc` のテキストパースと HTTP 応答が主で、メモリ安全性に強い要件 (秘密鍵の取り扱い等) はない。Rust の利点が活きる場面ではない。
- **エコシステム適合**: `go-systemd` 等の systemd 連携ライブラリがあり、将来の unit チェック実装にも橋渡しが効く。

## Alternatives Considered

- **Rust**: warden 系 (`cache-warden`, `authsock-warden`) のメモリ管理・セキュリティ重視用途では採用しているが、本ツールは秘密情報を扱わない単純な HTTP I/O ツールであり、Rust の所有権モデル・コンパイル時間コストに見合う利点が薄い。学習・保守コストに対して得られるものが少ないため不採用。
- **Node.js / TypeScript**: 起動オーバーヘッドが大きく、メモリ使用量も Go より重い。Node ランタイムへの依存が配布性を下げる。`pkg` 等で単一バイナリ化しても Go 単体ビルドより複雑になる。
- **シェルスクリプト (bash)**: HTTP サーバを自前実装する手間 (`socat` や `netcat` 経由) が大きく、テスタビリティが低い。プロセス管理・並行性も非自明。常駐デーモンとしての安定運用に向かない。
