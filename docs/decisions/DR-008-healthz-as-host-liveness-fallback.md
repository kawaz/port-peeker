# DR-008: agent 自身の `/healthz` を `/check` と別に用意する

- ステータス: Accepted
- 日付: 2026-05-07

## Context

ヘルスチェックエージェントとしての本筋機能は `/check?port=N[&process=NAME]` であり、これだけで「指定ポートが LISTEN しているか」「指定プロセスが動いているか」を判定できる。しかし NLB / ALB のターゲットグループの中には、**「常時 LISTEN しているサービスがない、けれど死活は監視したい」** ケースがある。具体例:

- ACME `HTTP-01` 認証用の `:80` リスナー (例: `lego` standalone): 証明書発行・更新時のみ一時的に LISTEN し、普段は何も bind していない。`/check?port=80` を当てると常に `503 not listening` を返してしまい、ターゲットが恒常的に UNHEALTHY になる
- セキュリティ重要なポート (例: `sshd` の 22/22999/etc.): ホストの死活監視はしたいが、NLB のヘルスチェックで TCP 開閉を繰り返すと sshd の auth/connect 系ログがヘルスチェック起源で埋まる。本物の不正アクセスや異常がノイズに紛れ、インシデント検知の見逃しに直結しやすい

## Decision

agent 自身の死活確認専用エンドポイント `/healthz` を `/check` と別に用意し、**常に `200 OK\n` を返す**。port-peeker プロセスがリクエストを処理できる = ホスト全体が生きている、と読み替えられる。

ターゲットグループのヘルスチェックパスを目的に応じて使い分ける:

- **「特定ポートが LISTEN しているか」を判定したい** → `/check?port=N[&process=NAME]`
- **「ホスト全体が生きていれば良い (個別ポートの LISTEN は問わない)」** → `/healthz`

## Rationale

- **常時 LISTEN しないサービスへの対応**: ACME の `HTTP-01` のような断続稼働ポートに対しては「`/check?port=80` で生死を判定する」こと自体が要件と合わない。`/healthz` を当てれば「ホストが生きていれば healthy」を表現できる
- **セキュリティ重要ポートの保護**: sshd 等のヘルスチェック対象から外すことで、NLB のプローブ起因の接続ログが発生しなくなる。port-peeker のコア設計 (`/proc` 直読みで対象サービスにログを発生させない) と同じ思想を、agent 自身が応答する `/healthz` まで拡張した形
- **判断の分離**: 個別ターゲットグループ毎に「実 LISTEN を確認するか、ホスト死活で十分か」を LB 側設定だけで切り替えられる。port-peeker のフラグや config を増やす必要がない
- **実装コストがほぼゼロ**: `/healthz` は固定文字列を返すだけのハンドラで、メンテナンスコストは無視できる

## Alternatives Considered

- **`/check` だけで運用 (`/healthz` を持たない)**: 常時 LISTEN しない 80 番のターゲットグループや、sshd のようなヘルスチェック対象にしたくないポートに対する受け皿が無くなる。結果として LB 側で「このターゲットグループはヘルスチェック無効」とせざるを得ず、本物の障害検知も失う
- **ターゲットグループのヘルスチェックを無効化**: ヘルスチェック自体ができなくなり、ホストが落ちても LB が気づけない
- **`/check` に「ホスト死活モード」を相乗り (例: `/check?host=alive` のような代替パラメータ)**: 1 エンドポイントが「ポート個別判定」と「ホスト全体死活」の 2 用途を兼ねることになり、API としての意図がぼやける。`/healthz` が独立しているほうが、ヘルスチェックパスを見ただけで意図が読み取れる
- **sshd 等を直接ヘルスチェック対象にする (NLB の TCP プローブを当てる)**: ヘルスチェック起因の接続ログが大量発生し、本物の攻撃や異常を埋没させる。インシデント見逃しリスクのほうが、sshd 直チェックで得られる精度より重い
