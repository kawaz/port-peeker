# DR-007: ツール名を `port-peeker` とする

- ステータス: Accepted
- 日付: 2026-05-07

## Context

設計が固まり実装に着手する直前、ツール名を決める必要があった。本ツールのコア設計は「LB のヘルスチェックを実サービスポートに直接当てない」「`/proc` を読むだけで TCP 接続を発生させない」という点にあり、要するに **「直接繋ぐとうるさいので、こっそり覗いてだけ判定する」** ことが本質。命名はこのニュアンスを反映するべきだった。

命名の方針:

- 名前から「何をするツールか」が伝わる程度の自己説明性
- 短さは過剰に求めない (2 ワード合成は許容)
- 「こっそり覗く」のトーンが滲む語感
- GitHub のリポジトリ名としても発音・記述しやすい

## Decision

ツール名を **`port-peeker`** とする。バイナリ名・モジュール名・リポジトリ名・パッケージ名すべて統一。

## Rationale

- **コア設計との一致**: `peek` は「ちらっと覗く」「直接接近せずに見る」のニュアンスを持ち、`/proc` 経由で listening 状態を観察するという設計思想と意味的に一致している。`net.Dial` を使わない判断 (DR-002) と命名が同じ言葉で語れる
- **キャラ性**: `port-peeker` は「ポート覗き屋」という主体を名乗る形になっていて、ツール自身の役割が一語で立ち上がる。`peek-check` のような動詞形より、`port-peeker` のような名詞形 (= エージェント名) の方が、常駐デーモンとしての存在感に合う
- **自己説明的**: `port` と `peeker` の合成で、英語話者でなくても「ポートを覗き見るもの」と推測できる
- **2 ワード合成の利点**: 短さよりも分かりやすさを優先する方針 (1 ワードに圧縮する必要なし) なので、`port-peeker` のような自然な合成語が選択肢として有効。`peek` 単独だと汎用すぎて GitHub 等で衝突しやすいが、`port-peeker` まで具体化すれば名前空間も確保しやすい

## Alternatives Considered

- **動詞形 (`peek-check` / `proc-peek` / `health-peek`)**: 「peek してチェックする」という動作は伝わるが、ツール自身を主体として表現できず、エージェント名としては弱い
- **比喩系 (`pulse` / `vigil` / `lookout` / `sentry`)**: ヘルスチェック・監視のメタファーとしては綺麗だが、「何をするか」が一語からは推測しづらい。`pulse` `sentry` などは既存プロダクトとの衝突も気になる
- **機能直球系 (`portcheck` / `hcheck` / `lbhc`)**: 機能は明快だが「直接ではなく覗くだけ」という設計のニュアンスが消えてしまい、`net.Dial` 系の素朴な実装と区別がつかない
- **`peek` 単独**: 短すぎて GitHub・パッケージ名空間で衝突しやすい。検索性も悪い
- **ステルス・隠密系 (`shadow` / `phantom` / `stealth-check`)**: トーンが過剰で、運用ツールとしての真面目さと釣り合わない

## 命名トーンの今後への波及

将来サブツールやコンパニオンツールを作る場合、同じトーン (= 主体を名乗る名詞合成、ツールの役割が一目で分かる、過度に短くしない) で揃えることで、シリーズ全体としての一貫性を保つ。
