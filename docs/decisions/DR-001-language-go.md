# DR-001: 実装言語に Go を採用

- ステータス: Accepted
- 日付: 2026-05-06

## Context

port-peeker の責務は「LB のヘルスチェック HTTP リクエストを受け、`/proc` を読んで結果を返す」だけのワンバイナリツールである。実装言語を確定させる必要があった。

## Decision

**Go** を採用する。

## Rationale

- **HTTP サーバ系は標準ライブラリだけで完結する**: `net/http` で `/check` と `/healthz` を書くだけなら依存ゼロで成立する。Go の HTTP / TCP / TLS 周りは枯れていて、ヘルスチェックエージェントの全要件を標準だけで満たせる。
- **静的リンク + クロスコンパイルが容易**: `CGO_ENABLED=0 GOOS=linux GOARCH=amd64 go build` で macOS から Linux バイナリが作れる。配布バイナリが 1 ファイルで完結する。
- **メモリフットプリントが小さい**: 10MB 未満で常駐可能。
- **責務は I/O 中心**: `/proc` のテキストパースと HTTP 応答が主で、秘密情報を扱わずメモリ安全性に強い要件はない。

## Alternatives Considered

- **Rust**: 同じ程度の HTTP サーバを書こうとすると非同期ランタイム (`tokio`) や HTTP クレート (`axum` / `hyper`) など、標準外の依存が必須になる。本ツールに必要なのは軽量な HTTP サーバ + `/proc` 読み + プロセス常駐だけで、その規模に対して所有権モデル・コンパイル時間・依存数のトレードオフが釣り合わない。
- **Node.js / TypeScript**: 起動オーバーヘッドとメモリ使用量が重く、Node ランタイムへの依存が配布性を下げる。
- **シェルスクリプト (bash)**: HTTP サーバを自前実装する手間が大きく、テスタビリティ・並行性・プロセス管理いずれも常駐デーモンに向かない。
